DIFFICULTY // 到底有多难？

以京东 h5st 签名为例——网站为了防止自动化访问，会给每个请求加一个"签名"。生成这个签名的 JS 代码经过了层层防护，每多一层，逆向难度就翻倍。

代码混淆
所有变量名被替换成无意义的字符串，函数被层层嵌套包装，控制流被打乱成跳转表。人眼读起来就是一堆乱码。
JD 实际代码：
function _0x3a7f(_0x2b1c) {
  var _0x4e8d = _0x2b1c[_0x5f9a(0x1a3)];
  return _0x4e8d ? _0x2b1c[_0x5f9a(0x1b7)] : _0x3e2f;
自定义虚拟机
代码不是直接执行的，而是被编译成自定义"字节码"，由一个解释器逐条读取运行。等于在程序里面又套了一层程序。
JD 实际情况：
发现 5 个 独立的 VM 字节码解释器
每个 VM 有自己的指令集和操作码
花了 40 轮（R38-R78）才逐个击破
多层加密嵌套
一个签名不是简单的"加密一次"。数据经过多次变换——先排序、再拼接、再哈希、再编码……中间任何一步理解错了，最终结果就全错。
JD h5st 签名结构：
10 段 分号分隔
每段由不同算法独立生成
段与段之间有依赖关系
输出格式: ts;fp;appId;tk;h1;v;t;c;h2;a
反调试与环境检测
网站会检测你是否在用开发者工具、是否在模拟浏览器环境。一旦检测到异常，加密函数会返回错误结果——看起来像正常工作，但输出是错的。更狡猾的是，有些代码会检测屏幕分辨率、Canvas 指纹等，确保运行环境是"真实的浏览器"。
体积与复杂度
JD 的签名算法藏在一段 241KB 的混淆 JS 文件中。这不是 241KB 的普通代码——其中大量是死代码、陷阱分支、字符串加密函数。真正的核心逻辑被埋在几千行"垃圾"中，像大海捞针，而且针还会伪装成海水。
这些防护不是单独出现的——它们组合叠加。混淆 + VM + 多层加密 + 反调试 + 大文件，使得人工逆向需要极高的专业能力和大量时间投入。
这类任务人能不能做？能做，但非常耗时耗力——需要资深专家投入大量时间反复调试、验证、踩坑、重来。
而 AI Agent 系统的价值在于：把"专家花两周反复调试"的事，变成"一次启动、自动完成"——7x24 不知疲倦、多角色交叉验证、85 轮迭代自动收敛。
THE TEAM // AI 团队

8 个专业 Agent，各司其职

为什么需要 8 个 Agent，而不是一个大 Agent 全包？
就像医院不会让一个医生同时负责诊断、化验、拍片、审核、开药——每个环节需要不同的专业视角和独立的验证逻辑。特别是"魔鬼代言人"（Adversary），它的职责就是推翻其他 Agent 的结论——如果分析者和验证者是同一个人，就失去了交叉验证的意义。
还有一个关键原因：上下文隔离。每个 Agent 启动时都是全新的上下文空间，只接收当前任务的指令和必要的背景信息。如果把所有工作塞进一个大 Agent，85 轮对话积累的海量细节会互相干扰——前面的推测被当成事实、过时的假设影响新判断、关键证据在上下文压缩时被丢弃。独立 Agent 天然避免了这些"记忆污染"问题。

INFRASTRUCTURE // 支撑层

这个流程能跑起来，底层依赖三样东西——缺一不可

MCP
Model Context Protocol
AI 的"手脚"
让 AI 能连接真实系统——控制浏览器（注入代码、读日志、点击页面）、读写文件、执行脚本。没有 MCP，AI 只能输出文字，什么都做不了。
Skill
可复用工作流模板
AI 的"SOP 手册"
把专业领域的方法论固化成模板——定义流程步骤、调度规则、验收标准。一次编写，所有同类任务都能复用。你正在看的这个逆向系统就是一个 Skill。
Agent
专业化角色分工
AI 的"专家团队"
每个 Agent 扮演一个专业角色，有独立的职责边界和验证逻辑。分析者和验证者分离，确保结论经过交叉检验而非一家之言。
PROTOCOL // 协作规范

它们怎么配合？

8 个 Agent 像团队里的 8 个专家，它们之间不需要开会，靠 4 套"制度"来协调——谁该做什么、做完怎么交、结果放哪，都有明确规范。

知识图谱
Knowledge Graph
类比：共享作战地图
所有 Agent 共享同一张"地图"——记录已知结论、证据、待解决问题。只有指挥官（PM）有权限更新地图，其他 Agent 只能读取。
claims 结论主张（已验证 / 待验证 / 已推翻）
evidence 支撑证据（工具 + 数据来源）
agent_history 执行记录（谁做了什么、何时）
classification 分析分类（黑盒类型、推荐策略）
任务指令单
pm_instruction
类比：PM 给每个专家发的任务单
每轮开始，指挥官给目标 Agent 发一份指令单——"这轮你要做什么、上下文是什么、做完产出什么文件"。Agent 只做指令单里要求的事。
goal 本轮具体目标
round 当前轮次编号
previous_summary 上一轮摘要
expected_outputs 预期产出文件清单
forced_actions 必须执行的动作
工作报告
Summary File
类比：专家完成后的汇报模板
每个 Agent 做完后，必须写一份固定格式的报告。不允许口头汇报，必须写到文件。PM 阅读报告来决定下一步调度谁。
1. 本轮目标 — 指令单的 goal
2. 执行步骤 — 按时间记录工具调用
3. 推理过程 — 观察 → 推导 → 排除 → 结论
4. 关键证据 — 原始数据 + 来源标注
5. 验收判定 — 标准对比实际结果
6. 产出文件 — 生成的文件列表
7. 下一步建议 — 建议 + 理由 + 风险
文件系统
Workspace
类比：共享项目文件夹
Agent 之间不直接对话，全部通过文件系统传递信息。每个目录有严格的读写权限——谁可以放什么文件，有明确规定。
workspace/project/task/
  input/      原始JS源码（禁止写入）
  output/     分析结果、验证脚本（所有Agent）
  overrides/  注入文件（仅Injector）
  KG.yaml    知识图谱（仅PM）
LIFECYCLE // 协作循环
PM 更新知识图谱 → PM 发任务指令单 → Agent 执行任务 → Agent 写工作报告 → PM 读报告，决定下一步 → 循环...

整个逆向过程就是这 5 步的循环——直到知识图谱中所有结论都被验证，达到目标等级

🔒
权限隔离
每个 Agent 只能写自己负责的目录。Injector 不能动 input，Analyzer 不能动 overrides
📄
文件即通信
Agent 之间不对话、不传消息。所有信息交换通过读写文件完成——可追溯、可审计
🔐
统一日志规范
注入代码使用统一的 Key 体系和数据格式，确保任何 Agent 都能正确解读采集到的日志

HIGHLIGHTS // 核心亮点

为什么这套系统令人震撼？

自验证闭环
像学术论文的同行评审——但更严格。密码学家写论文，质检员复核，然后专门的"魔鬼代言人"会主动试图推翻结论。三者都通过，结论才被接受。

知识图谱自动积累
一个自动生长的"作战地图"。每一轮分析都往地图上添加新发现、修正旧假设、标注哪些路走不通。所有 Agent 共享同一张地图。

递归拆解
像拆俄罗斯套娃——当你破解了一个锁，发现钥匙本身也被锁住了。系统会自动把那个新锁作为新的逆向目标，继续破解。

诚实分级
AI 不说"我做到了"就完事——它给自己打分：A+ = 独立客户端验证通过，A = 算法复现成功，B/C/D = 还需要越来越多人帮助。
JD: 主动识别出 5 个 VM 字节码差距阻止了真正 Level A，花 40 轮逐个关闭
端到端自动化
从"帮我分析这个目标"到"这是一个能独立工作的 API 客户端"——全程无需人工干预。人只需要启动任务和审查最终结果。

GET STARTED // 如何开始？
1
找任务
在自己的工作中找一个重复性高、步骤固定的任务——每次做都要走一样的流程、查一样的资料、写一样的文档。
2
拆 SOP
把这个任务拆解成清晰的步骤：每一步做什么、输入什么、输出什么、怎么判断做对了。写成文档——这就是你给 AI 的工作说明书。
3
封装 Skill / Agent
把 SOP 配置成 Skill（可复用的工作流模板）或 Agent（专职角色）。之后每次同类任务，一条命令启动，AI 按你的流程执行。
从 探索 AI 到 系统性运用
不需要等 AI 变得完美，也不需要学会写代码。
你最擅长的事情是理解业务、定义流程、判断结果——
把这些经验交给 AI 去执行，这就是 MCP + Skill + Agent 的意义。
